# フェーズ管理基盤実装

## 実装完了日
2025-12-27

## 概要
Battleシーンのフェーズ管理システムを実装しました。道中→ボス→シナリオへの遷移フローが制御でき、Timelineの静的参照と動的生成の両方に対応しています。

## アーキテクチャ

### Model層（フェーズ定義・管理）

| クラス名 | 役割 | 特徴 |
|---------|------|------|
| `BattlePhaseSequenceAsset` | フェーズ一覧ScriptableObject | 1つのシーケンス（道中/ボス）が複数フェーズを保持 |
| `BattlePhaseDefinition` | フェーズの設定 | phaseId/時間制限/Timeline/終了条件を保持 |
| `BattlePhaseModelBase` | フェーズの抽象基底 | Enter/Exit/OnExitPhaseイベント |
| `TimeLimitBattlePhaseModel` | 時間制限の実装 | UniRx.Observable.Timerで制限時間後に自動終了 |
| `BattlePhaseSequenceModel` | フェーズ遷移管理 | 現在のフェーズを追跡、`TryMoveNext()` で順番に進む |
| `BattlePhaseSequenceRepository` | シーケンス読込 | Addressablesからロード＆キャッシュ |

### Presenter層（フロー制御・Binding）

| クラス名 | 役割 | 特徴 |
|---------|------|------|
| `BattleScenePresenter` | ゲームフロー管理 | ステージ解決→道中再生→ボス再生→シナリオ遷移 |
| `BattlePhaseStateMachine` | Timeline再生制御 | PlayableDirector の asset差し替え＆Binding適用 |
| `BattleTimelineBindingMap` | Track→Object対応表 | track名ベースで自動Binding |

### Timeline動的生成

| クラス名 | 役割 | 特徴 |
|---------|------|------|
| `BattleTimelineBuilderAsset` | Timeline生成エンジン | Signal/AnimationTrackをデータから構築 |
| `SignalTrackDefinition` / `SignalEmitterDefinition` | Signal構造体 | time＋SignalAssetでマーカーを作成 |
| `AnimationTrackDefinition` / `AnimationClipDefinition` | Animation構造体 | 既存AnimationClipをstart/durationで配置 |

### その他

| クラス名 | 役割 |
|---------|------|
| `ScenarioTransitionRequest` | シナリオ遷移リクエスト（ステージ番号/ID/タイミング） |

## 主要フロー

### ゲーム開始から終了まで

```
BattleScenePresenter.InitializeAndStart()
  ↓
StageModel(RuntimeModel から取得)
  ↓
WaySequenceAssetをロード → BattlePhaseStateMachine.PlaySequence()
  ↓
フェーズ1再生 → 終了時刻に完了 → フェーズ2へ...
  ↓
道中完了 → RequestScenario(scenarioIdWayToBoss) 発火
  ↓
[シナリオシーン挟む]
  ↓
CompleteScenarioTransition() → BossSequenceAsset ロード
  ↓
ボスフェーズ再生（同様に遷移）
  ↓
ボス完了 → RequestScenario(scenarioIdBossToNext) 発火
  ↓
[シナリオシーン挟む]
  ↓
CompleteScenarioTransition() → stageModel.Clear() → OnBattleCompleted 発火
```

### フェーズ内Timeline再生

```
BattlePhaseStateMachine.ApplyTimeline(phase)
  ↓
phase.ResolveTimeline()
  ├─ 静的TimelineAsset の場合 → そのまま返す
  └─ BattleTimelineBuilder の場合 → BuildTimeline() で動的生成
  ↓
playableDirector.playableAsset = resolvedTimeline
  ↓
BattleTimelineBindingMap.ApplyBindings(director, timeline)
  ├─ timeline.outputs を列挙
  ├─ track名で bindings を検索
  └─ SetGenericBinding() で紐づけ
  ↓
playableDirector.Play()
```

## Timeline動的生成の仕組み

### 使い方例

**BattleTimelineBuilderAsset の設定**
```
Signal Tracks:
  - TrackName: "SignalReceiver"
    Emitters:
      - Time: 0.5, Signal: BulletSpawnSignal
      - Time: 1.5, Signal: EnemySpawnSignal

Animation Tracks:
  - TrackName: "Boss"
    Clips:
      - Start: 0, Duration: 2, AnimationClip: BossWalk
      - Start: 2, Duration: 1, AnimationClip: BossAttack
```

実行時に `BuildTimeline()` を呼ぶと：
1. `TimelineAsset` をメモリに作成
2. SignalTrack を "SignalReceiver" という名前で作成→Emitterを配置
3. AnimationTrack を "Boss" という名前で作成→Clipを配置
4. 返却されたTimelineをPlayableDirectorに設定

フェーズ終了時に `Dispose()` で自動削除（`HideFlags.DontSave`）

## Bindingの設定方法

### 準備手順

1. **BattleSceneで**
   - `BattlePhaseStateMachine` が付いているGameObject を用意
   - そこに `PlayableDirector` をアサイン
   - 同じGameObject に `BattleTimelineBindingMap` コンポーネント追加

2. **BattleTimelineBindingMapで**
   - `bindings` リストに要素を追加
   - `Track Name`: Timeline上のトラック名（完全一致）
   - `Target`: 対象GameObject/Component

3. **Timeline/Builderで**
   - SignalTrack/AnimationTrack の名前を上記と一致させる

### 例

```
Track Name = "Boss"          → Target = BossのAnimator
Track Name = "SignalReceiver" → Target = SignalReceiver付きGameObject
```

## 次の実装タスク

### 必須

1. **シナリオ遷移の配線** [優先度: 最高]
   - `BattleScenePresenter.OnScenarioRequested` を購読
   - Scenarioシーンを開く
   - Scenario終了時に `BattleScenePresenter.CompleteScenarioTransition(timing)` 呼び出し
   - どのシーンに戻すか（Battle継続 or 次ステージ）の導線確定

2. **Signal処理の実装**
   - 各Signal種類（弾発射/敵出現など）を `SignalAsset` として定義
   - `SignalReceiver.OnXxxSignal()` メソッド群を実装
   - Receiver → Manager/StagePresenter への委譲

3. **ステージごとのデータ投入**
   - 各ステージ用 `BattlePhaseSequenceAsset` を作成
   - 静的Timeline or Builder の選択＆設定
   - Stage1/2/3... の StageData に address を入力
   - Addressables登録＆アドレス確定

### 推奨

- フェーズ終了条件の追加（HP閾値、敵全滅検知など）
- ポーズ/リトライ時の `BattlePhaseStateMachine.Stop()` 呼び出し設計
- Runtime Timeline のメモリリーク検証＆対応

### 後回しOK

- 道中・ボス用に別々のSignalReceiver構成
- ControlTrack等の動的生成対応
- ファイナライザ等の詳細な リソース管理最適化

## 設計判断・トレードオフ

### なぜ「Presenterにはmodel wrapper を通す」か
- Presenterアセンブリが Infra (StageData) に依存するのを避け、アセンブリ境界を保つ
- 後で Infra側を変更しても、Presenter層への波及が最小で済む

### なぜ「Track名ベースのBinding」か
- 静的Timeline と 動的生成Timeline の両方に対応しやすい
- Timeline側のTrack名を「何にバインドするか」が分かる名前にするだけで、コード変更不要

### なぜ「HideFlags.DontSave」か
- Runtime生成されたTimelineはシーン保存時に含めない
- ゲーム起動時に毎回生成され、メモリは自動管理される
