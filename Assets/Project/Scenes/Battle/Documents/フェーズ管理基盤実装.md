# フェーズ管理基盤実装

## 実装完了日
2025-12-27（最終更新: 2026-01-14）

## 概要
Battleシーンのフェーズ管理システムを実装しました。道中→ボス→シナリオへの遷移フローが制御でき、Timelineの動的生成に対応しています。

## アーキテクチャ

### Model層（フェーズ定義・管理）

| クラス名 | 役割 | 特徴 |
|---------|------|------|
| `BattleSequenceAsset` | フェーズ一覧ScriptableObject | 1つのシーケンス（道中/ボス）が複数フェーズを保持。ファイル名でType自動検出 |
| `BattlePhaseDefinition` | フェーズの設定 | phaseId/時間制限/Builder/終了条件を保持 |
| `BattlePhaseModelBase` | フェーズの抽象基底 | Enter/Exit/OnExitPhaseイベント |
| `TimeLimitBattlePhaseModel` | 時間制限の実装 | UniRx.Observable.Timerで制限時間後に自動終了 |
| `BattleSequenceModel` | フェーズ遷移管理 | 現在のフェーズを追跡、`TryMoveNext()` で順番に進む |
| `BattleTimelineBuilderAsset` | Timeline生成エンジン | 6種類のトラックをデータから構築 |
| `EnemySpawnSignal` | 敵生成用カスタムSignal | spawnPosition/maxHpを保持、動的生成対応 |

### Presenter層（フロー制御・Binding）

| クラス名 | 役割 | 特徴 |
|---------|------|------|
| `BattleScenePresenter` | ゲームフロー管理 | ステージ解決→道中再生→ボス再生→シナリオ遷移 |
| `BattlePhaseStateMachine` | Timeline再生制御 | PlayableDirector の asset差し替え＆Binding適用 |
| `BattleTimelineBindingMap` | Track→Object対応表 | track名ベースで自動Binding |
| `Stage1SignalReceiver` | Signal受信処理 | INotificationReceiverでSignal種別を判定し処理を分岐 |

### Timeline動的生成（対応トラック）

| 定義クラス | クリップ/エミッター | 用途 |
|-----------|-------------------|------|
| `SignalTrackDefinition` | `SignalEmitterDefinition` | 汎用イベント発火 |
| `AnimationTrackDefinition` | `AnimationClipDefinition` | アニメーション再生 |
| `ActivationTrackDefinition` | `ActivationClipDefinition` | GameObject表示/非表示 |
| `AudioTrackDefinition` | `AudioClipDefinition` | 音声再生 |
| `ControlTrackDefinition` | `ControlClipDefinition` | Prefab生成・制御 |
| `EnemySpawnTrackDefinition` | `EnemySpawnDefinition` | 敵生成（Signal動的生成） |

## 主要フロー

### ゲーム開始から終了まで

```
BattleScenePresenter.InitializeAndStart()
  ↓
StageModel(RuntimeModel から取得)
  ↓
WaySequenceAssetをロード → BattlePhaseStateMachine.PlaySequence()
  ↓
フェーズ1再生 → 終了時刻に完了 → フェーズ2へ...
  ↓
道中完了 → シナリオシーンへ遷移
  ↓
[シナリオシーン挟む]
  ↓
BossSequenceAsset ロード
  ↓
ボスフェーズ再生（同様に遷移）
  ↓
ボス完了 → シナリオシーンへ遷移 → OnBattleCompleted 発火
```

### フェーズ内Timeline再生

```
BattlePhaseStateMachine.ApplyTimeline(phase)
  ↓
phase.CreateTimeline()
  └─ BattleTimelineBuilder.BuildTimeline() で動的生成
  ↓
playableDirector.playableAsset = createdTimeline
  ↓
BattleTimelineBindingMap.ApplyBindings(director, timeline)
  ├─ timeline.outputs を列挙
  ├─ track名で bindings を検索
  └─ SetGenericBinding() で紐づけ
  ↓
playableDirector.Play()
```

### Signal処理フロー

```
Timeline再生
  ↓
SignalEmitter発火 → Stage1SignalReceiver.OnNotify()
  ├─ EnemySpawnSignal の場合 → HandleEnemySpawn()
  └─ 汎用SignalAsset の場合 → HandleSignal()（名前で分岐）
```

## Timeline動的生成の仕組み

### 使い方例

**BattleTimelineBuilderAsset の設定**
```
Signal Tracks:
  - TrackName: "SignalReceiver"
    Emitters:
      - Time: 0.5, Signal: SampleSignal

Animation Tracks:
  - TrackName: "Boss"
    Clips:
      - Start: 0, Duration: 2, AnimationClip: BossWalk
      - Start: 2, Duration: 1, AnimationClip: BossAttack

Activation Tracks:
  - TrackName: "Effect"
    Clips:
      - Start: 1.0, Duration: 0.5

Audio Tracks:
  - TrackName: "BGM"
    Clips:
      - Start: 0, AudioClip: BattleBGM

Control Tracks:
  - TrackName: "Bullet"
    Clips:
      - Start: 0.5, Duration: 2.0, SourceObject: BulletPrefab

Enemy Spawn Tracks:
  - TrackName: "EnemySpawner"
    Spawns:
      - Time: 1.0, SpawnPosition: (0, 0, 5), MaxHp: 100
      - Time: 2.5, SpawnPosition: (3, 0, 5), MaxHp: 150
```

実行時に `BuildTimeline()` を呼ぶと：
1. `TimelineAsset` をメモリに作成
2. 各トラック定義から対応するTrackを作成・配置
3. EnemySpawnTrackは `EnemySpawnSignal` を動的生成してSignalTrackに配置
4. 返却されたTimelineをPlayableDirectorに設定

フェーズ終了時に `Dispose()` で自動削除（`HideFlags.DontSave`）

## Bindingの設定方法

### 準備手順

1. **BattleSceneで**
   - `BattlePhaseStateMachine` が付いているGameObject を用意
   - そこに `PlayableDirector` をアサイン
   - 同じGameObject に `BattleTimelineBindingMap` コンポーネント追加

2. **BattleTimelineBindingMapで**
   - `bindings` リストに要素を追加
   - `Track Name`: Timeline上のトラック名（完全一致）
   - `Target`: 対象GameObject/Component

3. **Timeline Builderで**
   - 各TrackDefinition の名前を上記と一致させる

### 例

```
Track Name = "Boss"           → Target = BossのAnimator
Track Name = "SignalReceiver" → Target = Stage1SignalReceiver付きGameObject
Track Name = "BGM"            → Target = AudioSource
Track Name = "Effect"         → Target = 表示/非表示したいGameObject
```

## 設計判断・トレードオフ

### なぜ「Presenterにはmodel wrapper を通す」か
- Presenterアセンブリが Infra (StageData) に依存するのを避け、アセンブリ境界を保つ
- 後で Infra側を変更しても、Presenter層への波及が最小で済む

### なぜ「Track名ベースのBinding」か
- 動的生成Timeline に対応しやすい
- Timeline側のTrack名を「何にバインドするか」が分かる名前にするだけで、コード変更不要

### なぜ「HideFlags.DontSave」か
- Runtime生成されたTimelineはシーン保存時に含めない
- ゲーム起動時に毎回生成され、メモリは自動管理される

### なぜ「EnemySpawnTrack」は専用定義か
- SignalAssetを動的生成してプロパティを設定する必要があるため
- 通常のSignalTrackではspawnPosition/maxHp等のパラメータを持てない
